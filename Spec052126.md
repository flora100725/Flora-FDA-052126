1. System Architecture & Core Topology
The system is designed on an event-driven, microservices-oriented architecture to separate compute-heavy AI orchestration from real-time client interaction. The core topology separates concerns into three decoupled tiers: the In-Browser Interactive Client Layer, the Agentic Orchestration & Orchestration Layer (Middleware), and the Scalable LLM & Document Transformation Pipeline.

1.1 In-Browser Interactive Client Layer
The client interface is a single-page application (SPA) optimized for low latency. It maintains a dual-state representation of the regulatory submission: the raw file layout and a structured, interactive JSON representation. The client establishes a continuous connection via WebSockets to receive step-by-step diagnostic and execution telemetry.
1.2 Agentic Orchestration & Execution Middleware
The middleware contains the execution core. It compiles high-level procedural instructions (from skill.md or a customized user prompt) and configurations (from agents.yaml) into a Dynamic Directed Acyclic Graph (DAG) of processing steps. This layer abstracts agent actions into a standardized state machine, managing prompt injections, agent-to-agent message queues, and memory structures.
1.3 Scalable Document Transformation Pipeline
This pipeline processes asynchronous file uploads. It utilizes structural parsing techniques to convert PDFs, CSVs, TXT documents, MD files, and JSON payloads into a unified regulatory data representation. The converted data is cached and indexed in an in-memory vector store to facilitate semantic lookups by the reviewing agents.
2. Multi-Format Data Ingestion & Transformation Pipeline
The platform supports multiple input formats. It converts unstructured and semi-structured documents into a unified, validated internal schema known as the Unified Medical Device Submission Schema (UMDSS).

2.3 Bi-directional Serialization Engine
The platform provides a dual-direction serialization engine. If a user modifies submission parameters inside the interactive web editor, the modifications flow through a translation engine. This engine synthesizes the modified UMDSS into exportable formats:
JSON Export: Outputs the updated structural object schema.
PDF Export: Passes the structural object into a templating engine (HTML-to-PDF) designed with standard regulatory headers, page indices, and timestamps.
CSV Export: Flattens structured risk management and testing data into standard tables.
Markdown/TXT Export: Serializes the hierarchy into markdown headers and structured paragraphs.

3.2 Visual Performance & Diagnostic Indicators ("Wow" Visual Indicators)
To provide real-time feedback during submission review, the frontend includes three live telemetry and status indicators:
Submission Completeness Gauge (Interactive Scorecard): A circular ring indicator that recalculates a completeness score based on the filled fields in the schema and the depth of evaluation text in each section. This indicator uses color coding to show current status: red (0-49%), orange (50-79%), and green (80-100%).
Live Agent State Telemetry Matrix: A visual grid displaying the activity state of each configured agent. Each cell represents an agent and changes color to indicate its operational phase:
Pulse Gray: Idle, waiting for context.
Pulse Blue: Agent active, loading contextual data.
Shimmering Amber: Agent calling LLM API (measuring latency in real time).
Steady Green: Execution finished successfully.
Pulsing Red: Fault state or validation failure.
Real-Time Token & Network Telemetry Dashboard: A live monitor graphing model consumption statistics, including tokens consumed, context window usage, processing velocity, and API roundtrip latency.

# Medical Device Premarket Review Guidelines (Skills Compilation)

## Core Directive: Regulatory Target Harmonization
All agents must perform checks to confirm that device descriptions support the chosen regulatory class (e.g., TFDA Class II or US FDA 510(k)).

## Skill Domain: Administrative Ingestion Checklist
1. Verify the applicant name exists and has registered business signatures.
2. Confirm the classification level conforms to target regions.
3. Check date parameters: For regional filings in Taiwan, convert and align Minguo calendar formats (e.g., year 115) to Gregorian formats (e.g., 2026).

## Skill Domain: Software & Cybersecurity Audit
1. If the device contains software, check if the software level of concern is declared (Major, Moderate, or Minor).
2. Validate that cybersecurity mitigations address standard attack vectors, such as unauthorized firmware modifications and sniffing of telemetry feeds.
3. Compare cybersecurity practices against FDA Cybersecurity in Medical Devices guidelines (Draft & Final Guidance recommendations).

## Skill Domain: Biocompatibility Auditing
1. Confirm compliance with ISO 10993-1.
2. Ensure that any contact parts list their material composition, duration of contact (e.g., limited, prolonged, permanent), and contact category.
4.3 Agent Coordinator Implementation Specs
The AgentCoordinator processes the review run. It performs the following orchestration logic:
code
Code
Algorithm: ExecuteAgenticReview
Input: UMDSS Object (S), Configuration File (agents.yaml), Skill Profile (skill.md), Target Language (Lang), Select LLM (Model)
Output: Comprehensive Review Report (R), Streamed Live Logs (L)

1. Parse S to determine target fields.
2. Read agents.yaml and instantiate selected agents.
3. Read skill.md and compile relevant rules into system prompt extensions.
4. For each selected agent (A_i) in Parallel/Sequential loop (based on dependency constraints):
    a. Extract corresponding subset of S (Domain-specific fields).
    b. Retrieve semantic context from vector database using domain-specific embedding matches.
    c. Format the prompt:
        Prompt_Payload = Base_Agent_Instruction
                        + "System Skills Profile: " + Compiled_Skills
                        + "Target Submission Content: " + Serialized_Sub_S
                        + "Retrieved Context Clues: " + Vector_Context
                        + "Instructions: Analyze and output findings in " + Lang
    d. Set temperature and parameters matching A_i config.
    e. Send payload to Model (e.g., gemini-3.1-flash-preview).
    f. Receive model response, parse structure, extract findings, and stream progress logs to the WebSocket server.
    g. Capture execution latency, tokens consumed, and validation status.
5. Aggregate findings from all completed agents.
6. Compile the Comprehensive Review Report (R) structured into standardized sections: Executive Overview, Detailed Core Domain Critiques, Critical Gaps, and Required Actions.
7. Return R.

5.1 AI-Generated Regulatory Cross-Mapping & Harmonization Engine
Description: Submissions target specific regulatory regions (e.g., US FDA 510(k), Taiwan TFDA registration, or EU MDR). This engine automatically analyzes a draft submission and cross-references its contents against international standards. It flags sections where documentation for one market fails to meet the specific administrative or clinical requirements of another.
Operational Mechanism: The engine maps inputs to standard regulatory criteria. If a TFDA Class II submission lacks specific materials details required under Taiwan's Medical Devices Act, it provides direct, contextual feedback (such as "Missing local testing certificate for wireless components required by NCC and TFDA").
UX Representation: An interactive map highlighting global regulatory targets (TFDA, FDA, CE). Clicking a market overlay reveals a list of regulatory gaps and localized requirements for that region.
5.2 Synthetic Request-for-Additional-Information (RAI) Letter Simulator (Adversarial Simulation)
Description: This feature generates synthetic "Request for Additional Information" (RAI) letters, mimicking the official deficiency notifications issued by FDA lead reviewers or TFDA technical auditors. It provides developers with an adversarial review of their submission prior to formal filing.
Operational Mechanism: The system evaluates the submission using specialized prompt instructions that prioritize searching for logical inconsistencies, insufficient sample sizes in performance data, unverified biocompatibility classifications, or vague cybersecurity monitoring plans. It outputs a formatted document structured like a regulatory agency deficiency notice (e.g., "TFDA 補件通知" or "FDA Deficiency Letter").
UX Representation: A split-screen interface displaying the draft submission on one side and simulated reviewer challenges on the other. Users can click on a simulated deficiency to jump directly to the relevant field in the submission editor and revise it.
5.3 Dynamic Evidence-to-Claim Semantic Lineage Mapper
Description: Pre-market reviews often require verifying that marketing or clinical claims map directly to underlying test reports and raw validation data. This engine builds and displays a semantic lineage graph that links claims to their supporting evidence.
Operational Mechanism: The tool parses claims found in the "Intended Use" or "Indications for Use" sections. It runs semantic queries against safety reports, bench tests, biocompatibility assays, and clinical data files in the vector database. It calculates a "Confidence Linkage Score" based on how well the documented evidence supports each claim.
UX Representation: An interactive canvas diagram on the dashboard. Lines connect clinical statements to physical evidence coordinates in the source PDF/JSON. Hovering over a connector line displays the confidence score and key matching passages from the test records.

7.2 Diagnostic and Error Handling Scenarios
The system is built to handle common operational errors gracefully:

# Medical Device Regulatory Premarket Review Guideline (Master Skills Profile)

This document establishes the evaluation logic, reference standards, and review thresholds used by the system's analytical agents.

---

## 1. Domain: Administrative & Regional Conformance Strategy
### 1.1 Local TFDA Regulatory Adherence (Taiwan)
1. Verify all registration forms, label designs, and Instructions for Use (IFU) align with the **Medical Devices Act (醫療器材管理法)** in Taiwan.
2. Confirm the classification level conforms to regional regulations.
3. Check date formats: Any local documents or applications submitted in Taiwan must use the Minguo calendar format (e.g., Year 115) and be converted to Gregorian formats (e.g., 2026) for international records.
4. Verify that manufacturer documentation lists valid QMS and GMP certifications, referencing local establishment license records.
5. Confirm that any Chinese labeling draft includes required elements:
   - Traditional Chinese name of the medical device.
   - Regional registration license number placeholders.
   - Name and full address of the local authorized representative.
   - Manufacturing batch number, manufacturing date, and shelf life.

### 1.2 US FDA 510(k) Substantial Equivalence Rules
1. Check that the submission designates at least one cleared predicate device with corresponding regulatory identifiers (e.g., K-numbers).
2. Compare indications for use, fundamental design, technology, materials, and safety characteristics between the subject device and the predicate.
3. Flag any differences in technology or materials that introduce new potential safety risks, requiring corresponding test data in the validation sections.

---

## 2. Domain: Hardware, Safety, and EMC (IEC 60601 Series)
### 2.1 IEC 60601-1 General Electrical Safety Requirements
1. Confirm compliance with **IEC 60601-1** (Edition 3.1 or 3.2).
2. Verify isolation barriers, creepage distances, and air clearances are documented and mapped in a functional insulation diagram.
3. Check that touch, patient, and leakage currents do not exceed limits under normal and single-fault conditions.

### 2.2 IEC 60601-1-2 Electromagnetic Compatibility
1. Confirm compliance with **IEC 60601-1-2** (4th Edition or later).
2. Review test configurations to ensure the device was tested in its most active operational mode.
3. Check that ESD immunity levels match clinical expectations for the target environment (e.g., $\pm 8\text{ kV}$ contact, $\pm 15\text{ kV}$ air discharge for home healthcare environments).

---

## 3. Domain: Materials & Biocompatibility (ISO 10993 Series)
### 3.1 ISO 10993-1 Evaluation Framework
1. Confirm that all components with direct or indirect patient contact are clearly listed with material names, duration of contact (limited, prolonged, or permanent), and categorization.
2. Ensure biocompatibility reports cover essential tests for the contact profile:
   - Cytotoxicity (ISO 10993-5)
   - Sensitization (ISO 10993-10)
   - Irritation / Intracutaneous Reactivity (ISO 10993-23)
3. For devices with gas-path contact (e.g., CPAP), check that tests evaluate particulate matter emission, volatile organic compounds (VOCs), and extractables in gas-path components (ISO 18562 series).

---

## 4. Domain: Software Lifecycle & Cybersecurity Safety
### 4.1 IEC 62304 Software Life-Cycle Processes
1. Verify the designated software safety classification matches the level of concern (Major, Moderate, or Minor).
2. Audits must find evidence of:
   - Software development plans, architecture specifications, and detailed designs.
   - Code verification protocols and automated testing logs.
   - Configuration management plans and bug tracking procedures.
3. If Class B or C software logic is present, verify a watchdog timer or parallel safety architecture is documented to handle runtime exceptions.

### 4.2 Medical Device Cybersecurity Guidelines
1. Evaluate cybersecurity controls against current regulatory recommendations (e.g., US FDA 2023 Cybersecurity Guidance).
2. Confirm a Software Bill of Materials (SBOM) is present and lists all third-party libraries and dependencies.
3. Check security controls for key interfaces, including:
   - Secure boot and digital signature verification of firmware updates.
   - Encryption of stored data and transmitted data (minimum TLS 1.3 or equivalent).
   - API rate limiting, session termination, and secure authentication.

---

## 5. Domain: Risk Management & Performance Validation
### 5.1 ISO 14971 Risk Management for Medical Devices
1. Verify the presence of a complete risk file detailing the hazard identification, risk estimation, and control measures for all known risk scenarios.
2. Confirm that all mitigations map directly to performance testing data or clinical validation records.
3. Flag any unmitigated risks or single points of failure that lack hardware- or software-level failsafes.
