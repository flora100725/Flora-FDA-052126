# Skill Definition

## name
generate-510k-performance-testing-summary-word-zhTW

## description
產生「TFDA / FDA 510(k) Performance Testing Summary（性能/安全性測試彙總）」的**詳細審查報告 Word（.docx）**，並以審查者（reviewer）可快速掃讀的方式呈現：每個標準/測試項目均包含**是否適用（是/否）**、**不適用理由（Justification）**、**證據追溯（報告/文件編號）**與**審查結論**。

**⚠️此技能必須在以下情境一律觸發（刻意「推」以避免 under-trigger）：**
- 使用者提到要「做 Word 報告 / 生成 docx / 套用樣板 / 測試總表 / 審查報告 / 彙整測試」
- 使用者提到 **TFDA、FDA、510(k)、SE（實質等同性）、Performance Testing、Safety Testing**
- 使用者提供任何 **IEC/ISO/NEMA/ASTM** 清單、測試矩陣、查檢表或模板，並要求轉為正式文件
- 使用者要求新增或補強：**「是否適用/不適用理由」**、**差異分析**、**合規矩陣**、**審查者友善格式**
- 使用者表示產品為**純軟體、醫療影像軟體（如 DICOM/PACS）、SaMD、或無病患接觸之裝置**，需要合理化「60601 不適用」等說明
- 使用者要求建立**可直接納入 510(k) submission** 或 **TFDA 查驗登記**的測試彙總章節

**此技能的核心目的：**
把零散的測試/標準資訊，轉成一份可直接提交、可追溯、可審查的「Performance Testing Summary」Word 報告，降低 reviewer 反覆補件（AI request）機率。

## compatibility
- 輸出格式：Microsoft Word（.docx）
- 預設語言：繁體中文（台灣）；可切換英文/日文（依使用者要求）
- 適用法規/框架（依裝置類型自動選配）：
  - FDA 510(k)（含 Substantial Equivalence 文件習慣）
  - TFDA 查驗登記（技術文件彙總習慣）
  - IEC 60601 系列、IEC 62304、ISO 14971、ISO 10993、IEC 81001-5-1（如適用）、DICOM（如適用）
- 不需外部依賴；基於使用者輸入與內建審查邏輯產製文件
- 允許輸入：文字、表格、清單、既有報告摘要、標準名稱、裝置描述、predicate 資訊

# Skill Behavior（技能行為）

## A. 輸入蒐集與結構化（Intake & Structuring）
1. 解析使用者提供的模板、標準清單或測試矩陣，標準化欄位（例如：標準名稱、版本、測試類別、報告名稱、文件編號、實驗室/出具單位、日期、適用產品）。
2. 依產品型態（純軟體/硬體/組合式/影像系統）建立「應適用標準候選集」並標註優先級：
   - 純軟體：強化 IEC 62304、ISO 14971、Cybersecurity、互通性（DICOM/PACS）等
   - 硬體/電氣醫療器材：優先 IEC 60601-1 / -1-2 / -1-8 等
   - 光/雷射：IEC 60825-1、IEC 62471 等
3. 將每個項目建立唯一追蹤 ID（例如：PT-001, PT-002…），便於審查追溯。

## B. 產生 Word 報告骨架（Report Skeleton）
自動產生 Word 報告，至少包含下列章節（可依使用者模板增刪）：
1. 封面與文件控制（Document Control）
   - 文件名稱、文件編號、版本、機密等級、出具單位、日期
2. 目的與範圍（Purpose & Scope）
3. 裝置概述（Device Overview）
   - 裝置類型、用途（Intended Use 摘要）、軟硬體邊界
4. 性能/安全性測試彙總表（Performance Testing Summary Tables）—**本技能核心**
5. 重要結論與缺口摘要（Key Findings & Gaps）
6. 附錄（可選）
   - 報告清單、縮寫、參考文件、變更紀錄（Revision History）

## C. 每項標準/測試的表格格式（Reviewer-Friendly Tables）
每個測試/標準項目建立一致表格（建議固定 8 欄，便於 reviewer 掃描）：
- 測試/標準項目（含版本/年份）
- 報告名稱
- 文件編號
- 出具單位（實驗室/內部部門/第三方）
- 適用產品/型號/軟體版本
- **是否適用（是/否）**
- **不適用理由（Justification）**（若適用則填「NA」或簡述適用依據）
- **審查結果（Review Result）**（Pass/Fail/NA + reviewer 備註）

> 規則：  
> - 「不適用」不得只寫「不適用」，必須給出可被審查接受的理由（例如：無病患接觸/無電氣硬體/非雷射/無輸出光譜/不具警報功能等）。  
> - 「適用」則必須有證據追溯（至少報告名稱 + 文件編號/連結/儲存位置）。

## D. 三個進階 AI 強化功能（每次都要啟用）
1. **自動缺口分析（Auto Gap Analysis）**
   - 依裝置類型與使用者提供內容，比對「可能需要但尚未提供」的測試/文件（例如：軟體層級、62304、風險管理、互通性測試、資安測試等），在「缺口摘要」列出建議補強項目與原因。
2. **證據追溯與一致性檢核（Evidence Traceability & Consistency Check）**
   - 檢查同一標準在不同段落是否出現矛盾（例如表格寫 NA 但內文提到已測），並提出修正建議。
   - 檢查文件編號/報告名稱的格式一致性，降低 reviewer 困惑。
3. **Reviewer 風險紅旗提示（Reviewer Risk Flags）**
   - 自動標記可能引發 FDA/TFDA 補件的表述（例如：暗示診斷/治療宣稱、含糊的「符合」但無證據、標準版本不明、適用性理由不足），並在備註區提供建議替代語句（偏法規措辭）。

## E. 針對「純軟體/影像軟體」的特化規則（Software-only / Imaging SaMD）
當裝置屬於純軟體或醫療影像軟體（DICOM/PACS）時，技能必須：
- 預設將 IEC 60601-1 / 60601-1-2 等硬體測試列為「可能不適用」，並提供合理化語句模板：
  - 例：本產品為獨立軟體，無病患接觸之電氣硬體；其安全性以軟體生命週期與風險管理流程評估為主。
- 主動提示應補強的常見項目（依情境顯示於缺口摘要）：
  - IEC 62304（軟體生命週期）
  - ISO 14971（風險管理）
  - IEC 81001-5-1 / FDA 資安期待（若有網路/互通性）
  - DICOM conformance statement、PACS 互通性測試（若有 DICOM 傳輸）
  - 影像顯示/處理一致性、資料完整性、輸出可重現性（性能驗證）
- 避免導向診斷宣稱：僅描述「顯示、處理、分析並提供結果供專業人員判讀」。

# Output Requirements（輸出要求）
- 必須輸出：單一 Word（.docx）檔
- 文件必須包含：
  1) 文件控制資訊（文件名稱/編號/版本/日期/出具單位）  
  2) 章節目錄或清楚的段落階層  
  3) 每個標準/測試項目的表格（含是否適用與不適用理由）  
  4) 缺口摘要（至少 5 點，若資料不足仍需以「待補資料」方式列出）  
  5) 結論（可提交性觀點：目前證據是否足以支持性能/安全性主張）

# Quality Bar（成功標準）
此技能只有在同時達到以下條件時才算成功：
- Reviewer 能在 30 秒內看懂：
  - 做了哪些測試、對應哪些標準、證據在哪裡
  - 哪些不適用、理由是否合理且一致
- 內容可直接納入：
  - FDA 510(k) submission 的 Performance Testing / Safety 相關章節
  - TFDA 技術文件之測試彙總
- 低模糊度：標準名稱、版本、文件編號、機構名稱一致且可追溯
- 對「純軟體」情境：不適用理由充分，並補出關鍵文件缺口（如 62304/14971/互通性/資安）

# Example Trigger Phrases（觸發範例）
- 「請把這份 IEC/ISO 測試清單做成 Word 審查報告」
- 「加上是否適用/不適用理由（Justification）」
- 「做 TFDA/FDA 510(k) Performance Testing Summary」
- 「這是純軟體，哪些 IEC 60601 不適用？請寫理由」
- 「我有 predicate device，請做測試彙總與差異說明」
- 「幫我把測試報告清單整理成 reviewer-friendly 表格」
